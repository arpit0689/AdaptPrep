ADAPTPREP PROJECT WORKFLOW AND FUNCTIONALITY DETAILS
====================================================

Project Name:
AdaptPrep

Project Type:
Full-stack MERN web application.

Purpose:
AdaptPrep is an intelligent adaptive exam preparation and consistency management platform for students preparing for Indian competitive and academic exams such as JEE, NEET, UPSC, GATE, SSC, CUET, CAT, and Board Exams.

It is not only a timetable app. It works like a study operating system that helps students plan topic-wise preparation, track consistency, recover from missed study days, manage backlog gradually, revise using spaced repetition, and stay motivated.

3. APPLICATION STARTUP FLOW
===========================

Frontend startup:
1. User runs frontend using Vite.
2. React app starts from frontend/src/main.jsx.
3. BrowserRouter wraps the app.
4. App.jsx defines public and protected routes.
5. ThemeProvider loads saved theme from localStorage or user preference.
6. Auth store checks localStorage for saved token/user.
7. If user is not logged in, protected pages redirect to /auth.
8. If user is logged in but onboarding is not completed, user is redirected to /onboarding.
9. If user is logged in and onboarding is completed, dashboard opens.

Backend startup:
1. server.js loads environment variables from backend/.env.
2. Express app is created.
3. Security middleware is registered:
   - helmet
   - cors
   - express.json
   - mongo sanitization
   - rate limiting
4. Health route is registered.
5. API routes are mounted.
6. Global not-found and error handlers are mounted.
7. MongoDB connection is started using MONGO_URI.
8. After database connection succeeds, Express starts listening on PORT.
9. Cron schedulers start for attendance, recovery, revisions, and weak area refresh.


4. AUTHENTICATION WORKFLOW
==========================

Main files:
- frontend/src/pages/AuthPage.jsx
- frontend/src/store/authStore.js
- frontend/src/services/api.js
- backend/routes/authRoutes.js
- backend/controllers/authController.js
- backend/models/User.js
- backend/middleware/auth.js

Register flow:
1. User opens /auth.
2. User switches to Register mode.
3. User enters name, email, and password.
4. Frontend sends POST /api/auth/register.
5. Backend validates:
   - name
   - email
   - password
6. Backend checks if email already exists.
7. Backend creates user in MongoDB.
8. User password is hashed using bcrypt before saving.
9. Backend creates JWT token.
10. Backend returns token and public user profile.
11. Frontend stores token in localStorage as adaptprep_token.
12. Frontend stores user in localStorage as adaptprep_user.
13. User is redirected to onboarding.

Login flow:
1. User enters email and password.
2. Frontend sends POST /api/auth/login.
3. Backend finds user by email and includes password field.
4. Backend compares password using bcrypt.
5. If password is correct, backend creates JWT token.
6. Frontend stores token and user in localStorage.
7. If onboarding is complete, user goes to dashboard.
8. If onboarding is incomplete, user goes to onboarding.

Protected route flow:
1. Frontend checks auth store.
2. Axios attaches JWT token to every request:
   Authorization: Bearer token
3. Backend protect middleware verifies token.
4. If token is valid, req.user is attached.
5. If token is invalid or missing, backend returns 401.
6. Frontend removes stale token on 401.

Logout flow:
1. User clicks Logout.
2. Frontend removes token and user from localStorage.
3. User is redirected to /auth.

Theme options:
- light
- dark
- system

How it works:
1. ThemeProvider checks localStorage for adaptprep_theme.
2. If no local theme exists, it uses user.preferredTheme.
3. If theme is system, browser prefers-color-scheme is checked.
4. The html element receives or removes the dark class.
5. CSS variables change colors globally.
6. When user changes theme, frontend:
   - updates local state
   - saves to localStorage
   - sends PATCH /api/settings with preferredTheme
7. Backend saves preferredTheme in user document.
8. Next login restores the same theme.

Dark theme:

- sage accents
- muted beige text
- gentle contrast

Light theme:
- warm off-white background

- sage accents

9. ROADMAP GENERATION WORKFLOW
==============================

Main files:
- backend/services/roadmapService.js
- backend/models/Task.js
- frontend/src/roadmap/RoadmapPage.jsx

Purpose:
Create topic-wise study tasks from exam syllabus and user preferences.

Input:
- exam date
- exam syllabus
- daily study hours
- weak subjects
- subject priorities
- study timing

Output:
Task documents with:
- subject
- chapter
- topic
- subtopic
- taskType
- duration
- plannedDate
- adjustedDate
- completed
- completionDate
- revisionStage
- notes
- priority
- sourceEngine
- difficultyLevel

Task types:
- Study
- Revision
- Mock Test
- Practice
- Recovery
- Weak Area Focus

How generation works:
1. Backend flattens syllabus into topic list.
2. Weak subjects receive higher priority.
3. Subject priority ordering affects task order.
4. Remaining days are calculated from today to exam date.
5. Daily study hours are split into realistic task blocks.
6. Tasks are distributed across future days.
7. Sundays or periodic days can include mock/reflection tasks.
8. Duplicate tasks are avoided using MongoDB unique indexes and bulk upserts.
9. TopicProgress records are created for every topic.

Roadmap page frontend:
1. User opens /roadmap.
2. Frontend fetches tasks for selected date.
3. Roadmap displays weekly board.
4. User can complete tasks from roadmap cards.
5. Refresh button reloads tasks.
6. Regenerate button calls backend regeneration endpoint.
7. Recover Tasks button activates recovery mode.

Regenerate button:
Frontend calls:
POST /api/tasks/regenerate

Backend:
1. Calls generateRoadmapForUser.
2. Creates missing roadmap tasks.
3. Avoids duplicate task creation.
4. Returns success message.

Recover Tasks button:
Frontend calls:
POST /api/tasks/recover

Backend:
1. Activates recovery mode.
2. Finds incomplete overdue tasks.
3. Redistributes them gradually into future dates.
4. Changes task type to Recovery where needed.
5. Creates supportive recovery notification.


10. TASK WORKFLOW
=================

Main files:
- backend/models/Task.js
- backend/controllers/taskController.js
- backend/routes/taskRoutes.js
- frontend/src/components/TaskCard.jsx

Task list:
GET /api/tasks

Task list by date:
GET /api/tasks?date=YYYY-MM-DD

Today focus:
GET /api/tasks/today-focus

Complete task:
PATCH /api/tasks/:id/complete

Skip task:
PATCH /api/tasks/:id/skip

Reschedule task:
PATCH /api/tasks/:id/reschedule

When user completes a task:
1. Frontend sends task completion request.
2. Backend finds task by ID and user ID.
3. Backend marks:
   - completed true
   - skipped false
   - completionDate current date
   - studyMinutesLogged
   - notes
   - difficultyLevel
4. Backend saves task.
5. Backend calls revisionService.
6. Revision tasks are generated after 1, 7, and 30 days.
7. TopicProgress is updated.

When user skips a task:
1. Backend marks skipped true.
2. Frontend shows supportive recovery message.
3. Future recovery can redistribute skipped/overdue work.

When user reschedules:
1. Frontend sends new date.
2. Backend updates plannedDate and adjustedDate.

13. SPACED REPETITION WORKFLOW
==============================

Main file:
- backend/services/revisionService.js

Purpose:
Improve retention by automatically scheduling revision tasks.

When topic task is completed:
1. scheduleRevisionsForTask is called.
2. Backend creates revision tasks:
   - after 1 day
   - after 7 days
   - after 30 days
3. Duplicate revisions are avoided by unique task index.
4. Revision task uses:

   - revisionStage 1, 7, or 30

=======================

Main files:
- frontend/src/attendance/AttendancePage.jsx
- backend/models/Attendance.js

1. User opens Attendance page.
2. User selects studied hours.
3. Frontend sends:
   POST /api/attendance
4. Backend compares studiedHours with planned daily hours.

Attendance status:

- Grace: studied hours >= 45% of planned hours
- Needs Attention: below grace threshold

Backend saves:

- dateKey
- studiedHours

- attendanceStatus
- focusScore
- source

Automatic attendance:
1. Cron job runs near end of day.
2. Backend checks completed tasks for that day.
3. Completed task minutes are converted to studied hours.
4. Attendance is automatically marked.

Attendance summary:
GET /api/attendance/summary

Returns:
- records
- attendance percentage
- weekly consistency score


15. STREAK WORKFLOW
===================

Main file:
- backend/services/streakService.js

Logic:
1. Attendance is marked.
2. streakService checks status.
3. Present or Grace counts as successful.
4. Successful day increases streak.
5. Needs Attention resets streak to 0.
6. longestStreak is updated if current streak is higher.
7. recoveryMode becomes true if attendance is not successful.
8. XP is increased for Present or Grace days.

Important:
The app uses dateKey to avoid duplicate same-day attendance logic.


16. RECOVERY MODE WORKFLOW
==========================

Main files:
- backend/services/recoveryService.js
- backend/services/roadmapService.js

Purpose:
Help user recover from missed tasks without guilt or overload.

Recovery mode activates when:
- attendance breaks
- user manually clicks Recover Tasks
- cron detects recoveryMode user

How it works:
1. Backend finds overdue incomplete tasks.
2. Tasks are not dumped into one day.
3. Backend calculates a daily recovery cap.
4. Overdue tasks are distributed over future days.
5. adjustedDate is updated.
6. taskType may become Recovery.
7. Notification is created with supportive message.

Supportive wording:
Instead of negative wording like "failed" or "behind", the app uses:
"Let's rebalance your roadmap gradually."

30. IMPORTANT NOTES
===================

1. Do not expose backend/.env publicly.
2. MongoDB Atlas IP access must allow your current network.
3. If MONGO_URI is missing, backend will not start.
4. If frontend cannot call backend, check VITE_API_URL.
5. If login expires, frontend removes stale token and asks user to login again.
6. Roadmap generation avoids duplicate tasks using indexes and upsert logic.
7. Recovery mode is designed to reduce overload and preserve motivation.
8. The current focus timer is frontend-only and can later be connected to attendance/study session tracking.
